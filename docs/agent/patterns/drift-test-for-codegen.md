# Drift test for every committed codegen output

## When to Use

Whenever a script writes a committed file from another committed source
(manifest → generated module, schema → types, SVG → React component).

## Implementation

Co-locate a `*.test.ts` next to the generator that:

1. Reads the source manifest / inputs.
2. Reads the committed generated file.
3. Re-runs the generator's deterministic transformation in memory (or
   asserts derived properties like sort order, completeness, signatures).
4. Asserts the two match exactly — same length, same content, same order.

Failure means a contributor edited the manifest without re-running the
build, or edited the generated file by hand. Both should fail CI.

## Examples

- `packages/icons/scripts/build-icon-data.test.ts` — verifies
  `src/icon-data.ts` is in lockstep with `src/icon-manifest.ts`. The
  canonical template for this pattern.
- `packages/icons/scripts/build.test.ts` — covers the SVGR pipeline's
  `src/svg-icons.ts` re-export.

The `.claude/ship-reviewed-prs-overrides.md` ("Generated / vendored —
DO NOT review") lists generated files reviewers should skip; the drift
tests are what makes that skip safe.

## Gotchas

- Mark the generated file with a header comment ("AUTO-GENERATED by
  scripts/build.ts — do not edit by hand"). Multiple files in this repo
  already follow this — keep doing it.
- The drift test must run as part of the package's normal `pnpm test`,
  not a special script. Otherwise CI doesn't catch the drift.
- If the generator depends on environment-dependent state (e.g. file
  ordering from `fs.readdir`), normalize it inside the generator so the
  drift test is deterministic.
- New generators should follow this template _immediately_ — don't ship
  the codegen without the drift test. Reviewers should reject any
  codegen PR that lacks one.
